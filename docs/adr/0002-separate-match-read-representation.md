# Separate the Match read representation from persistence

The Match read boundary will translate persistence records into a dedicated browser-facing representation before responding over HTTP. This keeps database naming, ORM models, and storage-specific values internal while allowing the browser contract to evolve deliberately; the representation's code name will be chosen when its responsibilities are fully settled. Both the completed-match and scheduled-match screens will consume this representation so Match has one browser-facing read shape. A Match Day and optional Kickoff Time are sent as separate plain-text values, rather than as a JavaScript date or timestamp. Match status selects a collection in the request but is omitted from each returned Match until a mixed-status screen needs it. The boundary also supplies its filter catalogue—years, months, teams, and leagues—so the browser has no manually maintained list of available values. The Match Results catalogue is based on finished Matches only, but its options remain independent rather than changing after each selection. The server validates filter values and rejects malformed requests rather than relying on browser-side validation. A well-formed search with no matching Matches is a normal empty result, not an invalid request.
